# Ambitionz Rebirth — Roadmap (27/06/2026)

Versão em produção: **v128_HOTKEYS** · `ambitionzgame.com` (deploy automático do `main` via Render).
Suíte rápida: **1446 testes verdes**. Soak de 100 partidas: **0 crash / 0 trava / 0 erro** (detalhes na §3).

---

## 1. Sumário executivo

O jogo é um TCG estilo Hearthstone **jogável de ponta a ponta**: o jogador cria conta, abre boosters, monta deck, joga duelos (arena/campanha/ranqueado), evolui/funde cartas, sobe de nível, ganha recompensas e acompanha o ranking. O motor de combate é **determinístico e auditável** (replay + verificação de paridade), a base de economia e progressão existe, e a camada de **feel/UX** (animações, áudio, atalhos) está madura.

O que falta para "AAA" não é a fundação — é **conteúdo** (cartas lendárias reais + arte), **profundidade competitiva** (PvP de verdade, eventos), **monetização ligada** (scaffolding pronto, desativado) e **polish guiado por jogadores reais** (telemetria + sua validação manual).

## 2. Estado atual — o que temos

### 2.1 Motor de jogo
- Pipeline determinístico (COMMAND→VALIDATE→BUILD_EVENT_STACK→RESOLVE_EFFECTS→REDUCER→EMIT→PERSIST) com **hash canônico de estado**, **replay** e **validação de paridade/invariantes**.
- Persistência SQLAlchemy: Postgres em produção, SQLite nos testes. Schema versionado (migrations).
- Bots de IA por perfil/dificuldade; campanha com chefes.

### 2.2 Modos e páginas
Arena (PvE), Campanha, Coleção, Deck Builder, Loja, Gemas (billing), Recompensas, Ranking, **Novidades/Patch Notes**, Perfil, Histórico, Suporte. APIs completas (start, play-card, attack, evolve, next-turn, mulligan, decks, catalog, crafting, ranking, async ghosts, billing).

### 2.3 Conteúdo
- **103 cartas**: 40 Comuns, 60 Incomuns, **3 Lendárias (placeholder, sem arte final, não craftáveis)**.
- 5 elementos/arquétipos (Fogo/Água/Terra/Sombra/Arcano) com keywords; guia de arquétipos no Deck Builder.
- Deck inicial de 30 cartas; famílias de duplicatas para evolução/fusão.

### 2.4 Economia e progressão
- Moedas **GOLD + COINZ** (carteira + ledger de auditoria/idempotência).
- **Forja/crafting**: desmanchar/criar Comum e Incomum via pó (lendárias fora por enquanto).
- **Boosters** (grátis no beta).
- **Níveis/XP**, **quests diárias/semanais** (`rebirth_retention.py`).
- **Temporada ranqueada** (ELO): recompensa por faixa (Bronze→Lendário) + soft-reset + bump, **gatilho manual** (`tools/ops/rebirth_season_reset.py`, idempotente por usuário/season).
- **Billing Stripe** (venda de gems): scaffolding presente, **desativado por flag** (`REBIRTH_ENABLE_BILLING`/`REBIRTH_ALLOW_STRIPE_LIVE`).

### 2.5 Competição
- **Ranking ELO** (começa em 1500; ajusta por partida).
- **Replays/ghosts verificáveis** (assíncrono, compartilhável por código) — PvE/replay-share por design enquanto anti-cheat não amadurece.

### 2.6 UX / Feel
- Rework visual da arena (mão menor, tabuleiro/slots maiores, layout adaptável à altura).
- Avatares circulares de herói (foto de perfil por presets, localStorage).
- **AV de combate maduro**: ~93 keyframes, screen shake contextual, números de dano flutuantes, reação do retrato ao dano, hit-pause, lunge, spark+flash, audio engine (sons por evento + volume por dano); **som + clarão de invocação** (v126).
- **Atalhos** (v127–v128): duplo-clique p/ jogar, drag-and-drop carta→altar, duplo-clique unidade→face, teclas 1–9, Espaço/Enter (encerrar), ESC / clicar-no-vazio (cancelar), feedback ao gastar mana.

### 2.7 Infra e qualidade
- Flask + Jinja + JS vanilla; **PWA + service worker** real em produção (cache-bust em 4 pontos por release).
- Deploy automático do `main` (Render). Wrapper **Android** com documentação de release (Play).
- **Telemetria + funnel** de produto; **1446 testes rápidos** verdes; padrão de QA "olhos-de-jogador" via Playwright (sem `force` + leitura de screenshots).

## 3. Saúde — resultado do soak de 100 partidas (27/06/2026)

Um bot Playwright jogou **100 partidas completas** como usuário (mulligan → invocar → atacar → encerrar → rematch), com `reduced_motion`.

| Métrica | Resultado |
|---|---|
| Partidas concluídas | 100/100 (todas com vitória/derrota definitiva) |
| Travamentos / jogos sem fim | **0 / 0** |
| Exceções / erros de console / página / HTTP 5xx | **0 / 0 / 0 / 0** |
| Tempo | 787s (~7,9s/partida) · média 12,1 turnos (4–25) |
| Winrate do bot | 11% |

**Leitura:** o **loop central de combate é robusto** — nenhum crash, estado preso ou erro em 100 ciclos completos, incluindo 100 rematches. O winrate de 11% é **artefato** de um bot intencionalmente burro (só invoca e ataca, sem estratégia), **não** um veredito de balance — balance se mede por telemetria de jogadores reais.

**Escopo do teste (honestidade):** o soak cobre o loop principal (invocar/atacar/encerrar/rematch, deck padrão, desktop). **Não** exercitou a fundo: fusão, magias/armadilhas, loja/crafting, deck-builder, campanha, mobile, nem os fluxos de vitória/recompensa (só 11 vitórias). "0 bugs" = "0 crashes no núcleo", não "livre de bugs em tudo".

## 5.1 Direção escolhida (27/06): profundidade competitiva

Decisão do dono: a próxima grande frente, **depois de fechar o P0**, é a **profundidade competitiva**. Ela aproveita o que já existe — **engine determinístico validado no servidor** (replay + paridade), **ranking ELO** e **ghosts verificáveis** — então a maior parte é construir por cima, não refazer. Sequência incremental, do menor risco ao maior:

1. **PvP assíncrono sobre decks** — ✅ **ENTREGUE (v129)**. Desafie o deck de outro jogador pelo Ranking ("Desafiar"); o servidor simula o duelo (o bot pilota o loadout do oponente) e o **ELO dos dois lados** é ajustado (espelho idempotente). Sem infra nova (roda no `-w 1 --threads 100`; match PvP vive no `MATCH_STORE` em memória). Verificado: 4 testes de ELO + e2e com 2 navegadores.
2. **Matchmaking ranqueado assíncrono** — ✅ **ENTREGUE (v129)**. "Partida ranqueada (PvP)" pareia por ELO próximo (`find_pvp_opponent`), cai pra PvE se o jogador está só. **Season automática**: o tool `tools/ops/rebirth_season_reset.py` está idempotente/cron-ready; ativar o cron (cadência + serviço) é decisão de infra/custo do dono — não auto-disparo (resetar ELO de todos é consequente demais).
3. **PvP em tempo real (dois humanos)** — 🟡 **BACKEND ENTREGUE; falta o cliente de frontend**. Roda por **polling no `-w 1 --threads 100`** (fila + partidas ao vivo em memória; **sem websocket/Redis → custo zero**). Engine (cirúrgico): `next_turn` pula a IA quando `_runtime_mode == "pvp_sync"`. `services/rebirth_live_pvp.py` usa o modelo **"ativo sempre no slot player" + troca de lados no handoff** → reusa `play_card`/`attack`/`next_turn` sem refatorar o core. Endpoints `/api/rebirth/pvp/live/{join,state,command}`: matchmaking, perspectiva por jogador, guarda de turno, **win→ELO simétrico** (liquidado em qualquer comando que finalize). Verificado: 4 testes unitários + e2e 2 navegadores (matchmaking, 80/80 trocas de turno, perspectiva, ataques causam dano, 0 erro). **Falta**: o **cliente ao vivo no arena** (modo polling: pollar estado, mostrar "vez do oponente", enviar via o endpoint live) — hoje o backend é jogável só via API. É build de UI, sem risco de engine.
4. **Torneios / ladder sazonal**: brackets + recompensas, sobre o matchmaking.

**Gates:** infra paga (websocket/escala) **não é necessária** pro PvP ao vivo via polling — então o gate é só esforço de engine, não custo. **Anti-cheat:** o engine já é server-validado (replay/paridade); a superfície é pequena.

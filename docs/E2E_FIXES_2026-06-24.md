# Correções dos achados de e2e/review (2026-06-24)

Resposta ao review externo que rodou `pytest -m e2e` (21 passed, 2 failed) e apontou
3 itens. Cache-bust `v118 → v119_ONBOARDING_FINALE`. Suíte rápida verde (1448).

## ✅ Corrigido

### #1 Onboarding — BURST furava o turno 1 protegido (flaky por seed)
- **Antes:** a mão da `guided-first-match` autenticada é semeada por usuário
  (`user:<id>:...`, [app.py](app.py:1701)); ~28% dos seeds abriam com um monstro
  BURST, que aplicava 2 de dano direto **na invocação** — furando a proteção de
  turno 1 (que só bloqueia ATAQUE direto) e deixando o teste de onboarding flaky.
- **Fix:** no primeiro duelo (tutorial), o dano de **face** do BURST é suprimido
  no turno 1 protegido ([rebirth_engine.py](services/rebirth_engine.py:636)). O
  monstro entra normalmente; só não detona na cara nesse turno. **Sem tocar o
  balance geral** (BURST segue normal fora do primeiro duelo).
- **Verificação:** `authenticated_first_turn` rodado **6× isolado → 6× pass**
  (era ~28% de falha). Determinístico agora.
- **Pendente (decisão de balance, por telemetria):** o mesmo furo existe no turno
  1 de partidas **normais** (ataque direto bloqueado, BURST não). Unificar afeta o
  meta de aggro FIRE/ARCANO BURST → fica como decisão do dono, não lab.

### #2 Finale mobile — action bar prendia o jogador
- **Antes:** com a partida encerrada, a `.rb-mobile-action-bar` (fixa no rodapé)
  interceptava o clique no CTA "Nova partida" no mobile.
- **Fix:** com vencedor definido (`#rebirth-board[data-winner]`), a action bar
  recebe `pointer-events: none` ([rebirth.css](static/css/rebirth.css)), liberando
  o CTA e os botões do finale. **Verificação:** `finale_overlay_does_not_block_new_match[mobile]` passa.

### #3 `playability_audit.py` agora é autônomo
- **Antes:** falhava sem `REBIRTH_DATABASE_URL`.
- **Fix:** liga `REBIRTH_ALLOW_SQLITE_TESTING` por conta própria (via `setdefault`,
  respeitando env real) antes de importar o app ([tools/playability_audit.py](tools/playability_audit.py)).
  **Verificação:** roda sem env nenhum → `RESULT=PASS`.

## ⚠️ Flake residual conhecido (infra de teste, NÃO regressão)
`authenticated_first_turn[mobile]` **passa 6/6 isolado**, mas na **suíte e2e cheia**
estoura no `expect_response("/api/rebirth/play-card")` (30s) — o clique de invocar
no mobile não dispara o request sob carga acumulada. Causa provável: o **server dev
é single-thread** e a fixture `live_server` é **session-scoped** (um processo Flask
para ~23 testes de browser); sob carga, requests concorrentes do mobile enfileiram.

**Por que não corrigi agora:** o fix de raiz seria `threaded=True` no `app.run`
dev, mas o SQLite de teste **não está configurado para threads**
(`check_same_thread=False` ausente) — arriscado rodar não supervisionado. Produção
usa WSGI (gunicorn), então não afeta o jogo.

**Recomendação (PR próprio):** tornar o SQLite de teste thread-safe + `threaded=True`
no server dev, OU dar à fixture `live_server` escopo por-teste para os testes de
interação mobile. A suíte **rápida** (gate do projeto) segue verde.
